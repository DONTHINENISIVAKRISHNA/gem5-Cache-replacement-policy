# gem5-Cache-replacement-policy
In my computer Architecture course , I worked on gem5 CPU Simulation, implementing a custom cache replacement policy called LRU_Variation on  an ARM O3 pipeline, in this project I Implemented LRU_Variation from a research paper on insertion and promotion vectors for last-level caches,a smarter eviction policy than standard LRU designed to reduce cache thrashing.
here there are two files, one is lru_variation.hh, defining the data structures and where lru_variation.cc implements the five functions, the key design decision was a shared position vector across all 16 lines in a set,when any line's rank changes, all other lines see the update immediately through the shared pointer. I implemented reset() to insert new lines at rank 10 instead of rank 0 — protecting proven hot lines from being displaced by one-time-access data. Touch() uses a promotion vector from the research paper,instead of always promoting to rank 0 on a hit, each rank maps to a specific target rank, so lines deep in the set get big jumps while lines near MRU get small jumps. GetVictim() evicts the line with the highest rank number ,the least recently used. Invalidate() sends lines to rank 15 guaranteeing immediate eviction.
"Running dijkstra_small on 32KB L1 and 128KB 16-way L2 I achieved IPC of 0.856, L1 data miss rate of 9.1%, and L2 data miss rate of 61.2%. The high L2 miss rate shows dijkstra's pointer-chasing access pattern exceeds the 128KB working set,the primary bottleneck was main memory latency. LRU_Variation's selective insertion reduced cold-line eviction of hot data compared to standard LRU , measurably improving IPC."
